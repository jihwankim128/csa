# CS Study Platform Business Document

## 1. 서비스 목적

CS Study Platform은 스터디원이 컴퓨터 구조, 운영체제, 네트워크, 데이터베이스 등 핵심 CS 주제를 글과 이미지 슬라이드로 정리하고, 리뷰 과정을 통해 신뢰도 높은 학습 콘텐츠를 축적하는 서비스다.

단순 글 모음이 아니라, 각 글이 주요 주제, 선행 주제, 연관 주제, 다음 학습 주제와 연결되어 사용자가 자연스럽게 다음 학습 경로를 탐색할 수 있도록 한다.

## 2. 핵심 가치

이 서비스의 핵심 가치는 다음과 같다.

- CS 개념을 글 단위가 아니라 주제 관계로 연결한다.
- 작성자는 Markdown 기반으로 빠르게 글을 작성한다.
- 그림이 필요한 부분은 이미지 슬라이드로 표현한다.
- 리뷰어는 글을 검토하고 수정 요청 또는 게시 승인을 할 수 있다.
- 사용자는 글을 읽으면서 선행 주제와 연관 주제를 함께 탐색할 수 있다.
- 서비스가 성장하면 주제 그래프, 로드맵, 임베딩 기반 추천, MCP 연동으로 확장한다.

## 3. 사용자 역할

### 3.1 작성자

작성자는 CS 주제에 대한 글을 작성한다.

작성자는 다음 작업을 수행할 수 있다.

- 글 작성
- 글 수정
- Markdown 미리보기
- 이미지 업로드
- 이미지 슬라이드 삽입
- 선행 주제 지정
- 연관 주제 지정
- 다음 학습 주제 지정
- 참고 자료 등록
- 리뷰 요청
- 리뷰 댓글 답변
- 본인이 작성한 글 목록 조회
- 상태별 작성 글 필터링

### 3.2 리뷰어

리뷰어는 리뷰 요청 상태의 글을 검토한다.

리뷰어는 다음 작업을 수행할 수 있다.

- 리뷰 대기 글 목록 조회
- 글 상세 확인
- 리뷰 댓글 작성
- 대댓글 작성
- 수정 요청
- 게시 승인

### 3.3 관리자

관리자는 서비스 운영에 필요한 기본 데이터를 관리한다.

MVP에서는 관리자 기능을 최소화한다.

관리 대상 후보는 다음과 같다.

- 사용자 권한
- CS 분야
- 주제
- 게시글 상태
- 부적절한 글 숨김 또는 삭제

## 5. 글 상태 생명주기

글은 다음 상태를 가진다.

```text
DRAFT
→ REVIEW
→ CHANGES_REQUESTED
→ REVIEW
→ PUBLISHED
```

### 5.1 DRAFT

작성자가 글을 작성 중인 상태다.

작성자만 글을 수정할 수 있다.

### 5.2 REVIEW

작성자가 리뷰를 요청한 상태다.

리뷰어는 글을 확인하고 리뷰 댓글을 작성할 수 있다.

작성자는 리뷰 댓글에 답변할 수 있다.

### 5.3 CHANGES_REQUESTED

리뷰어가 수정을 요청한 상태다.

작성자는 리뷰 내용을 반영하여 글을 수정할 수 있다.

작성자는 수정 완료 후 다시 리뷰를 요청할 수 있다.

### 5.4 PUBLISHED

리뷰어가 게시를 승인한 상태다.

공개 글 목록과 상세 페이지에서 조회할 수 있다.

게시 이후 수정 정책은 MVP에서는 단순화한다. 게시글 수정이 필요한 경우 작성자가 다시 수정 요청을 만들고, 리뷰어 검토 후 재게시한다.

## 6. 리뷰 정책

리뷰는 댓글과 대댓글 구조를 가진다.

리뷰 댓글은 특정 글에 속한다.

대댓글은 특정 리뷰 댓글에 대한 답변으로 사용한다.

MVP에서는 문장 단위 인라인 코멘트는 지원하지 않는다.

리뷰어는 다음 액션을 수행할 수 있다.

- 리뷰 댓글 작성
- 대댓글 작성
- 수정 요청
- 게시 승인

작성자는 다음 액션을 수행할 수 있다.

- 리뷰 댓글 확인
- 리뷰 댓글 답변
- 글 수정
- 재리뷰 요청

## 7. 주제 관계

글은 여러 주제와 연결될 수 있다.

주제 관계는 다음 타입을 가진다.

```text
MAIN
PREREQUISITE
RELATED
NEXT
```

### 7.1 MAIN

글의 핵심 주제다.

예시:

```text
"스레드는 무엇인가?" 글의 MAIN 주제는 "스레드"다.
```

### 7.2 PREREQUISITE

이 글을 읽기 전에 알면 좋은 선행 주제다.

예시:

```text
"스레드" 글의 PREREQUISITE 주제는 "프로세스", "메모리 공간"이 될 수 있다.
```

### 7.3 RELATED

같이 보면 좋은 연관 주제다.

예시:

```text
"스레드" 글의 RELATED 주제는 "경쟁 상태", "임계 영역", "상호 배제"가 될 수 있다.
```

### 7.4 NEXT

이 글을 읽은 뒤 다음으로 학습하면 좋은 주제다.

예시:

```text
"스레드" 글의 NEXT 주제는 "뮤텍스", "세마포어", "데드락"이 될 수 있다.
```

## 8. 글 추천 정책

MVP에서는 벡터 임베딩 기반 추천을 사용하지 않는다.

추천은 명시적으로 저장된 주제 관계를 기반으로 한다.

글 상세 페이지에서는 다음 정보를 보여준다.

- 선행 주제와 연결된 글 목록
- 연관 주제와 연결된 글 목록
- 다음 학습 주제와 연결된 글 목록

아직 게시된 글이 없는 주제는 MVP에서는 별도 추천하지 않는다.

빈 주제 탐색, 임베딩 기반 유사 글 추천, AI 기반 다음 주제 추천은 이후 단계에서 고려한다.

## 9. 이미지 슬라이드

본문은 Markdown 기반으로 작성한다.

이미지 슬라이드는 Markdown 확장 문법으로 표현한다.

예시:

```md
:::slider
![Step 1 - 프로세스 생성](/images/process-1.png)
![Step 2 - 스레드 생성](/images/thread-1.png)
![Step 3 - 공유 자원 접근](/images/shared-resource.png)
:::
```

렌더링 시 `slider` 블록은 이미지 슬라이드 컴포넌트로 변환된다.

## 10. MVP 범위

MVP에 포함한다.

- 로그인
- 글 작성
- Markdown 작성/미리보기
- 이미지 업로드
- 이미지 슬라이드
- 글 상태 전이
- 리뷰 댓글/대댓글
- 작성자 글 목록
- 작성자 글 상태 필터
- 리뷰어 리뷰 목록
- 게시 글 목록
- 게시 글 상세
- 주제 기반 추천

MVP에서 제외한다.

- 실시간 공동 편집
- 벡터 임베딩 추천
- 빈 주제 자동 탐색
- 그래프/마인드맵 UI
- MCP 연동
- 알림 시스템
- 좋아요/팔로우
- 문장 단위 인라인 리뷰 코멘트

## 11. 추후 확장 방향

### 11.1 검색 고도화

초기에는 PostgreSQL 기반 검색으로 시작한다.

글 수와 검색 요구가 증가하면 Meilisearch, Elasticsearch, OpenSearch 도입을 검토한다.

### 11.2 주제 그래프와 로드맵

주제 관계 데이터가 충분히 쌓이면 그래프 UI 또는 로드맵 UI를 제공한다.

로드맵은 선행 주제, 연관 주제, 다음 학습 주제 관계를 기반으로 구성한다.

### 11.3 임베딩 기반 추천

명시적 주제 관계만으로 추천 품질이 부족해지면 임베딩 기반 유사 글 추천을 도입한다.

초기 후보는 PostgreSQL과 pgvector 조합이다.

### 11.4 MCP 연동

외부 AI 도구가 프로젝트 데이터를 조회하거나 초안 작성을 도와야 할 때 MCP 연동을 검토한다.

초기 MCP 도구 후보는 다음과 같다.

- 글 검색
- 특정 글 조회
- 연관 주제 조회
- 다음 주제 추천
- 빈 주제 후보 탐색
- 글 초안 생성
